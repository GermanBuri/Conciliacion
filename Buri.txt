Etapa 1: Análisis funcional
1. Objetivo del sistema
La plataforma debe permitir a una empresa comparar de forma automática y asistida los movimientos de sus extractos bancarios contra sus registros contables, reduciendo trabajo manual, errores operativos, tiempos de cierre y riesgos de auditoría.

El sistema debe cubrir estos frentes:

ingestión de archivos bancarios y contables
validación y normalización de datos
conciliación automática por reglas
revisión manual asistida
gestión de diferencias y excepciones
trazabilidad completa de cada acción
reportes ejecutivos, operativos y de auditoría
2. Tipos de usuarios
Propongo estos perfiles iniciales:

Administrador global
Configura el sistema, empresas, seguridad, catálogos y parámetros generales.

Administrador de empresa
Administra usuarios de su empresa, cuentas bancarias, formatos y reglas de conciliación.

Analista de conciliación
Carga archivos, ejecuta conciliaciones, revisa pendientes, ajusta coincidencias y documenta diferencias.

Supervisor / Jefe financiero
Revisa resultados, aprueba conciliaciones, monitorea alertas y consulta reportes.

Auditor interno / externo
Consulta información histórica, evidencia, cambios, bitácoras y trazabilidad sin alterar datos operativos.

Esto ya marca una necesidad importante: seguridad multiempresa, RBAC y auditoría fuerte desde el inicio.

5. Reglas de negocio iniciales
Estas son reglas funcionales que conviene asumir desde ahora:

una conciliación siempre pertenece a una empresa, cuenta bancaria y período
un movimiento bancario no debe quedar conciliado dos veces, salvo reversión formal
un movimiento contable no debe quedar conciliado dos veces, salvo escenarios controlados de agrupación
toda conciliación manual debe registrar usuario, fecha y motivo
todo archivo cargado debe conservar evidencia de origen
los registros inválidos no deben entrar al motor sin pasar por validación
los cierres de período deberían poder bloquear edición posterior o exigir reapertura autorizada
las reglas de matching deben ser parametrizables por empresa o cuenta

Etapa 2: Arquitectura Empresarial
1. Decisión tecnológica principal
Backend: FastAPI
Elijo FastAPI sobre Django para este proyecto por estas razones:

enfoque API-first, ideal para separar backend y frontend profesionalmente
excelente soporte para tipado fuerte, validación y contratos claros
muy adecuado para arquitectura modular y servicios de procesamiento
documentación automática con OpenAPI/Swagger
mejor encaje para procesos asincrónicos, colas, workers y servicios de conciliación
menor acoplamiento que Django si queremos evolucionar a servicios especializados
Cuándo Django habría sido mejor
Django sería una buena opción si el foco principal fuera un backoffice monolítico con panel administrativo muy dominante desde el día 1. Pero en este caso el corazón del producto está en:

procesamiento de archivos
motor de conciliación
reglas configurables
UI moderna desacoplada
escalabilidad operativa
Por eso FastAPI encaja mejor.

2. Frontend recomendado
Recomendación: React + Next.js + TypeScript
Propongo:

Next.js como framework frontend
TypeScript para mantener contratos estables con el backend
Tailwind CSS para velocidad y consistencia visual
React Query / TanStack Query para manejo de estado servidor
Zustand o estado local simple para flujos UI
tabla avanzada con AG Grid o TanStack Table para conciliación manual
Motivo
La conciliación bancaria necesita interfaces con:

filtros complejos
tablas densas
comparaciones lado a lado
acciones masivas
experiencia fluida tipo aplicación empresarial
Eso se resuelve mucho mejor con un frontend desacoplado y rico que con templates server-side clásicos.

3. Arquitectura completa del sistema
Enfoque recomendado: Monolito modular bien diseñado
Para la primera versión empresarial, recomiendo un monolito modular y no microservicios desde el inicio.

Razón
Los microservicios demasiado pronto suelen introducir:

más complejidad operativa
observabilidad más difícil
más fricción en desarrollo
consistencia transaccional más compleja
En cambio, un monolito modular nos da:

velocidad de construcción
menor complejidad de despliegue
límites de dominio claros
posibilidad de extraer servicios después si el volumen lo exige
Componentes base
Frontend web: Next.js
API Backend: FastAPI
Base de datos: PostgreSQL
Cache / colas ligeras / locks: Redis
Procesamiento asíncrono: Celery o RQ
Workers: para importación, normalización, conciliación y generación de reportes
Object storage futuro: S3/MinIO para archivos si el volumen crece
Docker: desarrollo y despliegue consistente
Nginx o Traefik: reverse proxy
Logs / observabilidad: estructura preparada desde el inicio

9. Redis: rol recomendado
Redis no reemplaza la base de datos. Lo usamos como soporte operativo.

Usos principales
cola de tareas para procesamiento asíncrono
cache de consultas frecuentes
control de locks distribuidos
throttling y rate limiting
almacenamiento temporal de progreso de carga
sesiones efímeras si se requiere
idempotencia temporal en operaciones sensibles
Casos concretos
evitar procesar dos veces el mismo archivo simultáneamente
guardar estado “importando / normalizando / conciliando”
cachear dashboard ejecutivo
serializar tareas de conciliación por cuenta/período

12. Flujo de datos completo
Flujo principal
El usuario inicia sesión.
El frontend obtiene permisos y contexto de empresa.
El usuario carga archivo bancario o contable.
Backend registra metadatos del archivo y lo almacena.
Se dispara tarea asíncrona de parsing.
El parser interpreta el formato y genera registros staging.
El normalizador limpia y estandariza datos.
Si hay errores, se generan observaciones y se detiene confirmación.
Si pasa validación, los movimientos se consolidan en tablas operativas.
El usuario ejecuta conciliación para empresa/cuenta/período.
Se lanza worker del motor de conciliación.
El motor aplica reglas por prioridad.
Genera matches, diferencias, score y estados.
El frontend muestra resumen y pendientes.
El analista revisa y corrige manualmente.
El supervisor aprueba cierre si aplica.
Se genera reporte y se conserva auditoría completa.

16. Escalabilidad para miles de conciliaciones diarias
Enfoque general
Miles de conciliaciones diarias son totalmente viables si diseñamos bien estas piezas:

1. Procesamiento asíncrono
Las cargas, normalizaciones, conciliaciones y reportes no deben bloquear peticiones HTTP.

2. Particionamiento lógico
Aislar por:

empresa
cuenta bancaria
período
lote de importación
3. Índices correctos en PostgreSQL
Especialmente en:

empresa_id
cuenta_bancaria_id
fecha_movimiento
valor
referencia normalizada
estado_conciliacion
reconciliation_run_id
4. Batch processing
Procesar movimientos por lotes en vez de fila a fila.

5. Idempotencia
Evitar reprocesamientos accidentales de:

archivos
tareas
conciliaciones del mismo lote
6. Caching selectivo
No cachear transacciones sensibles sin criterio. Sí cachear:

catálogos
dashboard consolidado
conteos y métricas
7. Workers escalables horizontalmente
Poder subir más workers para:

parser
normalización
matching
generación de reportes
8. Separación futura si el volumen lo exige
A futuro se puede extraer el motor de conciliación como servicio independiente, pero no hace falta al inicio.

Reglas fuertes

no permitir cierre de corrida con diferencias abiertas si la política exige resolución
Recomendación técnica
No toda lógica debe vivir en constraints de BD. Parte debe vivir en capa de aplicación, porque:

algunas reglas son contextuales
algunas dependen del tipo de conciliación
otras pueden cambiar por política empresarial
18. Consideraciones para uno a uno, uno a muchos y muchos a uno
El punto clave aquí es que la conciliación no debe modelarse con columnas directas en movimientos, sino con una entidad intermedia.

Por eso reconciliation_results + tablas puente resuelven bien:

uno a uno: 1 banco + 1 contable
uno a muchos: 1 banco + N contables
muchos a uno: N bancos + 1 contable
muchos a muchos: lo dejamos soportado estructuralmente, aunque operativamente puede restringirse
Ese diseño evita rehacer la base cuando crezca la complejidad.

Etapa 4: Motor de Conciliación Inteligente
1. Objetivo del motor
El motor debe comparar movimientos bancarios y movimientos contables para:

identificar coincidencias automáticas con alta certeza
proponer coincidencias con confianza media
detectar diferencias y excepciones
soportar conciliación uno a uno, uno a muchos y muchos a uno
minimizar falsos positivos
permitir revisión manual controlada cuando no haya certeza suficiente
La regla principal aquí es simple: el motor debe ser útil, pero conservador. En conciliación bancaria empresarial, un falso match cuesta más que una coincidencia pendiente.

2. Principios de diseño
Principios base
determinístico primero, heurístico después
prioridad a exactitud sobre cobertura
explicabilidad obligatoria
configuración por empresa y por cuenta
reproducibilidad de resultados
aislamiento por período, empresa y cuenta
auditoría de cada decisión del motor
Implicación práctica
El motor no debe ser una “caja negra”. Cada match debe poder responder:

qué regla lo generó
qué campos coincidieron
qué tolerancias se usaron
cuál fue el score
por qué fue automático o quedó pendiente
3. Tipos de conciliación soportados
3.1 Uno a uno
Un movimiento bancario coincide con un movimiento contable.

Ejemplos:

transferencia por el mismo valor y fecha equivalente
consignación con referencia idéntica
comisión registrada una sola vez en ambos lados
3.2 Uno a muchos
Un movimiento bancario corresponde a varios movimientos contables.

Ejemplos:

un abono bancario que agrupa varias facturas
un pago bancario único contra varias partidas contables
un recaudo consolidado
3.3 Muchos a uno
Varios movimientos bancarios corresponden a un solo movimiento contable.

Ejemplos:

un asiento contable consolidado de varios ingresos
varias consignaciones aplicadas a un solo registro contable
3.4 Muchos a muchos
Estructuralmente puede existir, pero no recomiendo habilitarlo en automatización inicial. Puede quedar reservado para revisión manual avanzada porque su riesgo operativo es mucho mayor.

4. Pipeline del motor
Propongo este flujo interno del motor:

selección del universo
pre-filtrado
normalización operativa para matching
aplicación secuencial de reglas
resolución de conflictos
asignación de score
clasificación del resultado
generación de diferencias
persistencia y auditoría
flujo de revisión y aprobación
5. Selección del universo de conciliación
Antes de intentar hacer match, el motor debe acotar el conjunto de trabajo.

Filtros base
empresa
cuenta bancaria
período de conciliación
moneda
estado de conciliación pendiente o parcialmente conciliado
movimientos no reversados
movimientos no excluidos por regla
Resultado
Se forman dos conjuntos:

universo bancario candidato
universo contable candidato
Esto es clave para rendimiento y para evitar comparaciones innecesarias.

6. Pre-filtrado
Antes del scoring completo, conviene reducir combinaciones.

Criterios de pre-filtrado
signo compatible
débito con egreso
crédito con ingreso
valor dentro de tolerancia preliminar
fecha dentro de ventana máxima
coincidencia probable por cuenta contable asociada
referencia o texto con claves compartidas si existen
Beneficio
Esto reduce drásticamente el costo computacional y mejora precisión.

7. Estrategias de match
7.1 Match exacto
Es la estrategia de mayor confianza.

Criterios típicos
mismo valor neto
misma fecha exacta o fecha contable equivalente
misma referencia normalizada o muy alta coincidencia de referencia
signo compatible
moneda igual
Casos comunes
transferencias
consignaciones
pagos bancarios simples
comisiones registradas igual
Resultado esperado
Si cumple todos los criterios duros, el match puede marcarse como:

automatic_confirmed si la política lo permite
o suggested_high_confidence si requiere aprobación
7.2 Match por tolerancia
Se usa cuando hay pequeñas variaciones aceptables.

Tolerancias típicas
fecha: +/- N días
valor: +/- monto fijo
valor: +/- porcentaje
diferencia permitida por redondeo, retención o comisión conocida
Casos comunes
desfase entre fecha bancaria y contable
redondeos
diferencias menores autorizadas
registros contabilizados el día siguiente
Reglas
tolerancia de fecha configurable por empresa o cuenta
tolerancia de valor configurable por regla
si hay diferencia monetaria, debe quedar explícita
Resultado esperado
score medio o alto según combinación
si la diferencia económica supera umbral, no se confirma automáticamente
7.3 Match por referencia
Aquí la referencia o identificador de operación pesa mucho.

Campos relevantes
número de comprobante
referencia bancaria
consecutivo de transferencia
voucher
identificador externo
fragmentos numéricos útiles extraídos de texto
Proceso
normalizar referencia
extraer tokens
comparar coincidencia exacta o parcial
ponderar longitud, calidad y unicidad de la referencia
Cuidado importante
No toda referencia sirve igual. Una referencia como PAGO o TRANSFERENCIA no aporta valor real. El motor debe distinguir:

referencias de alta calidad
referencias débiles o genéricas
Resultado esperado
referencia fuerte puede elevar mucho el score
referencia débil no debe forzar match por sí sola
8. Match uno a muchos
Objetivo
Vincular un movimiento bancario con varias partidas contables cuyo total coincide.

Escenarios
un ingreso bancario agrupa varias facturas
un egreso bancario cubre múltiples comprobantes
recaudos consolidados
Estrategia
seleccionar candidatos contables por ventana de fecha y signo
agrupar por subconjuntos posibles
evaluar sumatoria contra monto bancario
validar referencias, tercero, descripción y proximidad temporal
escoger la combinación con mejor score y menor ambigüedad
Regla de negocio importante
No conviene abrir búsqueda combinatoria total sin control, porque explota en costo.

Restricciones recomendadas
máximo número de partidas por combinación: por ejemplo 2 a 5 inicialmente
ventana de fecha acotada
priorizar partidas con referencias relacionadas
cortar búsqueda cuando hay demasiadas combinaciones
Resultado
si la suma coincide exacta y el contexto es fuerte, puede proponerse con alto score
si hay varias combinaciones equivalentes, debe pasar a revisión manual
9. Match muchos a uno
Objetivo
Vincular varios movimientos bancarios a un solo asiento o movimiento contable.

Casos típicos
varias consignaciones registradas como un solo asiento contable
pagos fraccionados
recaudos parciales
Estrategia
La lógica es simétrica a uno a muchos:

seleccionar movimientos bancarios candidatos
generar combinaciones controladas
validar sumatoria contra movimiento contable
ponderar fechas, referencia, tercero y descripción
elegir la mejor combinación no ambigua
Restricciones
Las mismas:

límites de cardinalidad
ventana temporal
corte por complejidad
evitar elegir combinación si hay empate fuerte entre varias opciones
10. Reglas configurables
El motor debe ser gobernado por reglas, no por condicionales rígidos en código.

Tipos de reglas
por empresa
por cuenta bancaria
por tipo de movimiento
por banco
por rango de monto
por naturaleza débito/crédito
por origen del archivo
por patrón de referencia o descripción
Parámetros configurables por regla
prioridad
activa/inactiva
tipo de match permitido
tolerancia de fecha
tolerancia de monto
score mínimo para sugerencia
score mínimo para autoaprobación
obligatoriedad de referencia
obligatoriedad de descripción
límite de cardinalidad para uno a muchos
exclusiones
necesidad de aprobación manual
Ejemplo funcional de regla
nombre: Transferencias exactas cuenta principal
aplica a: cuenta bancaria X
tipo: exact
fecha tolerancia: 0
monto tolerancia: 0
requiere referencia: sí
score autoaprobación: 95
Otro ejemplo:

nombre: Consignaciones agrupadas
tipo: one_to_many
fecha tolerancia: 2 días
monto tolerancia: 0
máximo de partidas: 4
score autoaprobación: no permitido
requiere revisión manual: sí
11. Algoritmo de scoring
El score debe representar confianza, no solo similitud superficial.

Escala recomendada
0 a 100
Componentes del score
Propongo ponderar así, ajustable por regla:

monto: 35 puntos
fecha: 20 puntos
referencia: 25 puntos
descripción / tercero: 15 puntos
señales contextuales: 5 puntos
Total: 100

11.1 Score por monto
Criterio
coincidencia exacta: 35
dentro de tolerancia muy pequeña: 25 a 34
diferencia moderada permitida: 10 a 24
fuera de tolerancia: 0
Observación
El monto es una señal crítica. Si falla fuerte, normalmente no debe haber match automático.

11.2 Score por fecha
Criterio ejemplo
misma fecha: 20
diferencia de 1 día: 16
diferencia de 2 días: 12
diferencia de 3 días: 8
fuera de tolerancia: 0
Ajuste posible
Para fines de semana o cierres, algunas cuentas pueden usar una curva menos estricta.

11.3 Score por referencia
Criterio ejemplo
referencia exacta normalizada: 25
referencia con alta similitud: 18 a 24
coincidencia parcial de tokens relevantes: 10 a 17
referencia genérica o vacía: 0 a 5
Regla importante
Si una regla exige referencia y no existe coincidencia mínima, el match debe descartarse aunque otros factores sean altos.

11.4 Score por descripción / tercero
Criterio
similitud alta de descripción o tercero: 12 a 15
similitud media: 6 a 11
sin evidencia textual útil: 0 a 5
Técnicas esperadas después
normalización de texto
eliminación de stopwords operativas
similitud tokenizada
trigram o fuzzy matching controlado
11.5 Señales contextuales
Posibles señales
mismo banco/cuenta relacionada
mismo tipo de operación
mismo tercero esperado
patrón histórico compatible
regla prioritaria específica
Esto no debe dominar el score, solo desempatar o reforzar.

12. Clasificación por umbrales
Propongo esta política base:

95 a 100: match automático confiable
85 a 94: sugerencia fuerte, puede autoaprobarse según regla
70 a 84: sugerencia para revisión manual
50 a 69: coincidencia débil, mostrar solo como candidato secundario
0 a 49: no candidato
Importante
El score por sí solo no basta. También deben existir gates duros:

signo compatible
monto dentro de tolerancia máxima
unicidad suficiente
no conflicto con otro match más fuerte
13. Resolución de conflictos
Este punto es crítico. Puede haber varios candidatos para un mismo movimiento.

Casos de conflicto
un movimiento bancario coincide razonablemente con dos contables
varias combinaciones uno a muchos dan la misma suma
un mismo movimiento contable puede ser consumido por distintos matches
Estrategia recomendada
ordenar candidatos por score
aplicar prioridad de regla
preferir exact match sobre fuzzy
preferir menor diferencia de fecha
preferir referencia más fuerte
exigir diferencia mínima entre primer y segundo candidato
Regla de seguridad
Si el primer candidato no supera claramente al segundo, el resultado debe quedar en revisión manual.

Por ejemplo:

candidato A: 91
candidato B: 89
No debería autoaprobarse. Hay ambigüedad.

14. Orden de aplicación de reglas
El motor no debe ejecutar todo a la vez. Debe hacerlo por etapas.

Secuencia recomendada
exact match estricto
exact match con referencia fuerte
match por tolerancia de fecha
match por tolerancia de monto autorizada
match por similitud de referencia
uno a muchos
muchos a uno
sugerencias de baja confianza
generación final de diferencias
Beneficio
mejora rendimiento
evita consumir movimientos con reglas débiles antes de probar reglas fuertes
facilita explicabilidad

17. Conciliación manual asistida
El motor no termina donde falla. Debe asistir al usuario.

Qué debe ofrecer

Reducir trabajo manual sin forzar decisiones automáticas incorrectas.

20. Escalabilidad del motor
Para miles de conciliaciones diarias, el motor debe diseñarse con control de costo.

Estrategias
procesar por empresa + cuenta + período
prefiltrar antes del scoring completo
usar índices fuertes en monto, fecha, referencia y estado
ejecutar uno a muchos y muchos a uno solo sobre subconjuntos acotados
usar colas asíncronas
serializar corrida por cuenta/período con locks
guardar resultados intermedios si el lote es grande
permitir reprocess parcial sin recalcular todo
Muy importante
Los algoritmos combinatorios deben tener límites duros. Si no, el costo crece demasiado en agrupaciones.
